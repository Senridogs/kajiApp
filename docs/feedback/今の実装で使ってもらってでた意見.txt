ドラッグ&ドロップ実装案
前提: 現状の課題
現在のホーム画面（kaji-app.tsx:1933-2017）は「今日 / 明日 / 大仕事」の3セクションで、各セクション内のタスクは自動ソート（sortHomeSectionChores L162-219: 完了状態→担当者→アイコン→かな順）。ユーザーが並び順を変える手段がない。

また、タスクの実行日は lastPerformedAt + intervalDays で自動計算されるため、「この日にやる」を手動で変える手段もない。

ドラッグ&ドロップで実現する2つの機能
操作	効果	用途
セクション内ドラッグ	タスクの表示順を入れ替え	優先度を自分で決めたい
セクション間ドラッグ	タスクを別の日に移動	「明日やる」→「今日やる」等
ライブラリ選定
候補	評価
@dnd-kit	React向けDnDの定番。タッチ対応、ソート対応、軽量。推奨
motion (既存) の drag	単体のドラッグは得意だがソータブルリストは自前実装が必要。重い
自前実装 (touch events)	既存のスワイプ操作（タブ切替 use-swipe-tab.ts / 削除 use-swipe-delete.ts）との競合ハンドリングが極めて複雑
@dnd-kit を推奨。理由:

@dnd-kit/sortable でリスト内並び替えが簡単
DndContext + DragOverlay でセクション間移動もサポート
タッチセンサーに activationConstraint（長押し + 移動距離）を設定でき、スクロール・横スワイプとの干渉を避けられる
既存の motion アニメーションと共存可能
実装の全体像
A. データモデルの追加
// Prisma schema に追加
model ChoreScheduleOverride {
  id        String   @id @default(cuid())
  choreId   String
  fromDate  String   // 元の予定日 (YYYY-MM-DD)
  toDate    String   // 移動先の日 (YYYY-MM-DD)
  createdAt DateTime @default(now())
  chore     Chore    @relation(...)

  @@unique([choreId, fromDate])
}

または、既存の ChoreAssignment テーブルを拡張して scheduledDate フィールドを追加する案もある。既存の ChoreAssignment（schema.prisma:72-84）は choreId + date でユニークで、日付単位の担当者割り当てに使われている。日付の移動もこの仕組みに乗せることは可能だが、「担当者変更」と「日付移動」の意味が混ざるため、別テーブルの方がクリーン。

B. セクション内の並び替え（表示順のカスタマイズ）
方針: ユーザーが並び順を変えたら、その順番をローカルに保存（localStorage）。サーバー同期はオプション。

// 新規: ローカルの並び順管理
type SectionOrder = Record<string, string[]>; // { "today": [choreId1, choreId2, ...] }

// sortHomeSectionChores() の結果をユーザー定義順で上書き
function applyCustomOrder(chores: ChoreWithComputed[], order: string[]): ChoreWithComputed[] {
  if (!order.length) return chores;
  const indexMap = new Map(order.map((id, i) => [id, i]));
  return [...chores].sort((a, b) => {
    const ai = indexMap.get(a.id) ?? Infinity;
    const bi = indexMap.get(b.id) ?? Infinity;
    return ai - bi;
  });
}

メリット: サーバー不要、即座に反映
デメリット: 端末間で順番が共有されない
将来的にサーバー同期したければ、ChoreOrder テーブルを追加
C. セクション間の移動（日付の変更）
方針: タスクを「今日」から「明日」（またはその逆）にドラッグしたとき、ChoreScheduleOverride をAPIで作成。computeChore() と splitComputedChoresForHome() がオーバーライドを参照して正しいセクションに配置。

// ドロップ時の処理
async function moveChoreToSection(choreId: string, fromSection: string, toSection: string) {
  const fromDate = sectionToDateKey(fromSection); // "today" → "2026-02-16"
  const toDate = sectionToDateKey(toSection);      // "tomorrow" → "2026-02-17"
  
  // 楽観的更新: UIでは即座にセクション間を移動
  // APIでオーバーライドを保存
  await apiFetch('/api/chores/schedule-override', {
    method: 'POST',
    body: JSON.stringify({ choreId, fromDate, toDate })
  });
}

ジェスチャー	用途	競合リスク

実装面でのシンプルさ
自動スケジュールの計算はフロントエンドだけで完結できる：

function generateSchedule(chore: ChoreWithComputed, fromDate: Date, toDate: Date): Date[] {
  const dates: Date[] = [];
  // lastPerformedAtから intervalDays ごとに日付を生成
  let next = addDays(new Date(chore.lastPerformedAt), chore.intervalDays);
  while (next <= toDate) {
    if (next >= fromDate) dates.push(next);
    next = addDays(next, chore.intervalDays);
  }
  return dates;
}

問題の本質
一般的なタスク管理アプリが陥る罠:

  「やらなきゃ」が溜まる
      ↓
  開くたびに赤い⚠とか未完了の数字が目に入る
      ↓
  開くのが億劫になる
      ↓
  さらに溜まる
      ↓
  自己嫌悪 → アプリ削除

タスク管理アプリではなく「家事の相棒アプリ」にするのが目標。

家事の大変さ問題
重み付けはやらない方がいいと思います。

❌ 重み付けアプローチ:

  掃除機   = 2pt

  料理 30分、掃除機 10分...

  → 「手際が悪いから長いだけでしょ」で喧嘩
