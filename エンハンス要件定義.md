# エンハンス要件定義書（いえたすく v2）

## 1. 文書の目的

本書は「いえたすく」の現行実装（v1）を基盤とし、UX改善・機能追加を行うエンハンス開発の要件を定義する。
現行実装の調査結果および利用者フィードバックに基づき、**喧嘩の元にならない家事管理**という設計思想を軸に策定した。

---

## 2. 設計思想

### 2.1 基本方針

| 原則 | 説明 |
|------|------|
| 比較させない | 個人間の回数・割合を直接比較するUIを排除する |
| チームとして見せる | 統計は「おうち全体」単位で表示し、家族の一体感を出す |
| 感謝を促進する | 完了ログ・リアクション・メモを通じて「ありがとう」が生まれる設計にする |
| 操作を減らす | 通知設定・パートナー参加など初期設定の煩雑さを解消する |
| 誤操作を防ぐ | 取消など不可逆操作には確認を挟む |

### 2.2 やらないこと

| 項目 | 理由 |
|------|------|
| 家事への難易度・ポイント重み付け | 点数の妥当性自体が揉める原因になる。正解のない問題に正解を出そうとしない |
| 個人間の比較グラフ | 「私の方がやってる」論争の火種になる |
| ランキング・順位表示 | 競争ではなく協力の場にする |

## 3. 機能要件

### 3.1 統計画面の刷新

#### 3.1.1 現状の課題

現行の統計画面は以下の構成:
- **分担バランス**: ユーザー別の円グラフ（全タスク/通常/大仕事の3タブ切替）
- **各タスクごと詳細**: タスク別の棒グラフ（ユーザー別の色分け積み上げ）

これにより「〇〇さんは△回しかやってない」が一目で分かり、不満の可視化装置になっている。

#### 3.1.2 変更方針

**現行の統計画面（`StatsView`コンポーネント + `/api/stats`エンドポイント）を廃止**し、以下2つの画面に置き換える。

#### 3.1.3 おうちレポート（全員に表示）

統計タブを「おうちレポート」に変更する。

**表示項目**:

| セクション | 内容 | 目的 |
|-----------|------|------|
| 今月のおうち | 当月の家事完了総回数 + 先月比（±N） | チームとしての達成感 |
| よく回った家事 TOP3 | タスク別の完了回数上位3件（誰がは出さない） | 家の実態把握 |
| 久しぶりかも？ | 最終実施から一定期間（`intervalDays × 2` 以上）経過しているタスク一覧。最終実施日を付記 | リマインド目的。責めではなく気づき |

**期間切替**:
- 現行の週/月/半年/年/全期間/カスタムは廃止
- **直近3ヶ月分**を月単位で切替可能（「今月」「先月」「2ヶ月前」）
- デフォルト表示は「今月」
- 先月比は前月同期間との比較
- データは全期間保持（将来の年次総括等に利用可能）
- 3ヶ月より前のデータは表示しない（見れすぎるとプレッシャーになるため）

**非表示にする情報**:
- ユーザー別の回数・割合（円グラフ廃止）
- タスク別のユーザー内訳（積み上げ棒グラフ廃止）

#### 3.1.4 マイページ（本人のみ閲覧可能）

設定タブ内に「わたしの記録」セクションを追加する。

**表示項目**:
- 当月の自分の完了回数（タスク別内訳）
- 他の家族メンバーの数字は**表示しない**

**目的**: 自分の振り返り用。比較対象がないため純粋な達成感になる。

#### 3.1.5 API変更

**現行 `GET /api/stats`**:
- 現行レスポンス（`userCounts`, `bigTaskUserCounts`, `choreCounts[].userCounts`）はユーザー別集計を含む
- エンハンス後も内部的にはユーザー別集計を保持してよい（マイページ用）
- おうちレポート用に以下を追加:

```typescript
// 新規レスポンス型
type HouseholdReportResponse = {
  // 今月の合計
  currentMonthTotal: number;
  // 先月同期間の合計（比較用）
  previousMonthTotal: number;
  // タスク別回数（上位表示用、誰がやったかは含まない）
  choreCounts: Array<{
    choreId: string;
    title: string;
    icon: string;
    iconColor: string;
    bgColor: string;
    count: number;
  }>;
  // 久しぶりタスク
  staleTasks: Array<{
    choreId: string;
    title: string;
    icon: string;
    iconColor: string;
    bgColor: string;
    lastPerformedAt: string;    // ISO8601
    intervalDays: number;
    daysSinceLast: number;
  }>;
};

// マイページ用（セッションユーザーのみ）
type MyStatsResponse = {
  currentMonthTotal: number;
  choreCounts: Array<{
    choreId: string;
    title: string;
    icon: string;
    iconColor: string;
    bgColor: string;
    count: number;
  }>;
};
```

**エンドポイント設計**:
- `GET /api/household-report?month=2026-02` — おうちレポート（世帯全体集計）
  - `month` パラメータ: `YYYY-MM` 形式。省略時は当月
  - 直近3ヶ月分のみ許可（それ以前は400エラー）
- `GET /api/my-stats?month=2026-02` — マイページ用（セッションユーザーのみ集計）
  - 同じく `month` パラメータ対応
- 現行 `GET /api/stats` は基本廃止（移行期間中は併存、移行完了後に削除）

#### 3.1.6 「久しぶりかも？」の判定ロジック

```
対象条件:
  - archived = false
  - lastPerformedAt が存在する
  - daysSinceLast >= intervalDays × 2

表示順: daysSinceLast の降順（最も放置されているものが上）
表示上限: 5件
```

### 3.2 やったよログ（完了ログの共有表示）

#### 3.2.1 概要

**独立タブ「きろく」**として、世帯全体の直近の完了記録をタイムライン形式で表示する。

#### 3.2.2 現状の課題

現行では完了履歴は「リスト画面 → 各タスクの履歴ボトムシート」でしか確認できず、能動的に開かないと見えない。パートナーが何をやってくれたかが見えづらい。

#### 3.2.3 表示仕様

**配置**: メイン画面の独立タブ（ボトムナビの一つ）

**表示内容**:

```
きろく

  今日
    🍽 食器洗い ── ママ
      💬 "排水口もきれいにしたよ"
      ❤️ パパ
    🧹 掃除機  ── パパ

  昨日
    👕 洗濯    ── パパ
    🛁 風呂    ── ママ
      💬 "カビ取りもした！"
      👏 パパ
```

| 項目 | 仕様 |
|------|------|
| 表示件数 | 直近7日間、最大20件 |
| 表示情報 | タスクアイコン + タスク名 + 実施者名 + メモ（あれば）+ リアクション（あれば） |
| 日付グルーピング | JST日付でグループ化（「今日」「昨日」「2/15」等） |
| スキップ記録 | 表示しない（`isSkipped = true` を除外） |
| 初期記録 | 表示しない（`isInitial = true` を除外） |
| メモ表示 | メモがある場合のみ、記録の下に吹き出し風で表示 |
| リアクション表示 | リアクションがある場合のみ、メモの下に表示 |

#### 3.2.4 メモの位置づけ

現行ではメモは「通知に添えるメモ」という補助的な位置づけだが、やったよログ上でメモが常に見えることで**家族内の軽いコミュニケーション手段**になる。

- メモ入力UIの文言変更: 「通知に添えるメモを入力」→「ひとこと（任意）」
- メモのplaceholder変更: 「排水口もきれいにしたよ、など」

#### 3.2.5 データ取得

現行の `GET /api/records` をそのまま利用可能（直近200件取得済み）。
クライアント側で7日間分をフィルタリングして表示する。

#### 3.2.6 タブとしての独立理由

- ホーム画面下部に配置するとスマホで窮屈（ホームは今日/明日/大仕事で十分な情報量）
- 独立タブにすることで「きろくを見に行く」行為自体が家族への関心表明になる
- リアクション操作もタブ内で完結する

### 3.3 リアクション機能

#### 3.3.1 概要

完了記録に対して家族メンバーがリアクション（絵文字スタンプ）を送れる機能。

#### 3.3.2 目的

- 「ありがとう」「お疲れさま」を言葉にしなくても1タップで伝えられる
- やったよログ上でリアクションが見えることで、やった側のモチベーションになる

#### 3.3.3 仕様

**リアクション対象**: `ChoreRecord`（完了記録1件ごと）

**リアクション種別（固定プリセット）**:

| 絵文字 | 意味 |
|--------|------|
| 👏 | お疲れさま |
| ❤️ | ありがとう |
| ✨ | すごい！ |
| 🎉 | やったね！ |

- 1ユーザーにつき1記録あたり1リアクションまで（変更は可能、トグル式）
- 同じ絵文字をもう一度タップで取り消し
- 別の絵文字をタップで上書き

**表示場所**:
- やったよログの各記録行の下部
- 履歴ボトムシートの各記録行

**表示形式**:

```
🍽 食器洗い ── ママ
  💬 "排水口もきれいにしたよ"
  ❤️ パパ
```

リアクションがある場合のみ表示。絵文字 + リアクションした人の名前。

#### 3.3.4 データモデル

```prisma
model ChoreRecordReaction {
  id        String @id @default(cuid())
  recordId  String
  userId    String
  emoji     String   // "👏" | "❤️" | "✨" | "🎉"
  createdAt DateTime @default(now())

  record ChoreRecord @relation(fields: [recordId], references: [id], onDelete: Cascade)
  user   User        @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([recordId, userId])  // 1ユーザー1記録につき1リアクション
}
```

#### 3.3.5 APIエンドポイント

```
PUT  /api/records/:id/reaction   { emoji: string }   → リアクション追加/変更
DELETE /api/records/:id/reaction                       → リアクション取り消し
```

- リアクション追加時、**即時**で相手にプッシュ通知を送信（`notifyCompletion` 設定に準拠）
- `notifyCompletion = false` の世帯にはリアクション通知も送らない
- 通知内容: 「{ユーザー名} が ❤️ でリアクション」

#### 3.3.6 通知ペイロード

```typescript
{
  type: "reaction",
  title: "{ユーザー名} が {絵文字} でリアクション",
  body: "{タスク名}",
  url: "/"
}
```

### 3.4 完了取消の確認ダイアログ

#### 3.4.1 現状の課題

現行ではホーム画面のチェックマークを再タップすると**確認なしで即座に取消**される（`undoRecord`が直接実行される）。
誤タップで意図せず取り消してしまうリスクがある。

#### 3.4.2 変更内容

完了済みタスクの取消操作時に確認ダイアログを表示する。

**UI仕様**:

```
┌─────────────────────────────┐
│                             │
│  「食器洗い」の完了を        │
│   取り消しますか？           │
│                             │
│  ┌─────────┐ ┌──────────┐  │
│  │ やめる  │ │ 取り消す │  │
│  └─────────┘ └──────────┘  │
│                             │
└─────────────────────────────┘
```

| 要素 | 仕様 |
|------|------|
| 表示方式 | BottomSheet（既存コンポーネント流用） or インラインモーダル |
| タイトル | 「{タスク名}」の完了を取り消しますか？ |
| キャンセルボタン | 「やめる」— ダイアログを閉じる |
| 確定ボタン | 「取り消す」— `undoRecord()` を実行。赤系の色で警告感を出す |
| タップ外 | ダイアログを閉じる（キャンセル扱い） |

#### 3.4.3 対象操作

- ホーム画面でのチェックマーク再タップ（完了 → 未完了に戻す）
- 履歴画面からの取消（現行実装がある場合）

**注意**: タスクの「削除」（リスト画面でのスワイプ削除）は現行のUndoToast方式を維持する（5秒以内なら元に戻せる）。

### 3.5 通知UIの簡素化

#### 3.5.1 現状の課題

現行の設定画面では以下の通知設定が並んでおり、初見で何を設定すべきか分かりづらい:

```
現行:
  □ 期限当日通知      ← 何の通知？いつ来る？
  □ 期限超過通知      ← 「超過」って怖い
  □ 完了時通知        ← 誰の完了？
  [通知テスト]
  [Push通知 ON/OFF]   ← これと上の3つの関係は？
```

さらに、通知時刻（`reminderTimes`）はUI上に設定導線がなく、デフォルト値のまま変更できない。

#### 3.5.2 変更内容

通知設定を2段階に整理する。

**Step 1: 通知のON/OFF（大きなスイッチ）**

```
┌────────────────────────────────┐
│ 🔔 プッシュ通知                │
│                                │
│ タスクのリマインドや、パートナー│
│ が家事を完了したときにお知らせ  │
│ します。                       │
│                                │
│               [━━━━ ON ━━━━]   │
└────────────────────────────────┘
```

- これが OFF の場合、以下の詳細設定は全て非表示（グレーアウトではなく非表示）
- ON にした時点で `Notification.requestPermission()` を呼ぶ
- 許可が得られなかった場合はエラーメッセージを表示してOFFに戻す

**Step 2: 詳細設定（通知ONの場合のみ表示）**

```
┌────────────────────────────────┐
│ リマインド                     │
│ やるべき家事があるとき通知      │
│                     [━━ ON ━━] │
│                                │
│ 通知する時間                   │
│ ┌──────┐ ┌──────┐ ┌───┐      │
│ │ 8:00 │ │18:00 │ │ + │      │
│ └──────┘ └──────┘ └───┘      │
│                                │
│ パートナーの完了通知            │
│ パートナーが家事を完了したとき  │
│ 通知                           │
│                     [━━ ON ━━] │
└────────────────────────────────┘
```

| 現行設定 | 変更後 | 備考 |
|---------|--------|------|
| 期限当日通知 + 期限超過通知 | 「リマインド」に統合 | 分ける意味が薄い。どちらも「やるべき家事がある」通知 |
| 完了時通知 | 「パートナーの完了通知」に改名 | 誰の完了かを明示 |
| Push通知 ON/OFF | Step 1のメインスイッチに昇格 | 他の設定との関係を明確化 |
| 通知テスト | 詳細設定の末尾に配置 | 目立たなくてよい |
| 通知時刻（reminderTimes） | UIに露出させる | 現行はUI未露出でデフォルト固定 |

#### 3.5.3 通知時刻の編集UI

```
通知する時間
┌──────┐ ┌──────┐ ┌───┐
│ 8:00 │ │18:00 │ │ + │
└──────┘ └──────┘ └───┘
```

- 各時刻はチップ形式で表示。タップで削除（×マーク）
- 「+」ボタンでネイティブの時刻ピッカーを表示して追加
- 最低1つは必須（0個になる削除は不可）
- 最大4つまで

#### 3.5.4 データモデル変更

```
Household:
  notifyDueToday: Boolean       → 「リマインド」ON/OFFに統合
  remindDailyIfOverdue: Boolean → 「リマインド」ON/OFFに統合（廃止）
  notifyCompletion: Boolean     → そのまま維持（「パートナーの完了通知」）
  reminderTimes: String[]       → そのまま維持

新規フィールド:
  notifyReminder: Boolean @default(true)  // リマインド統合フラグ
```

**マイグレーション**: `notifyDueToday = true OR remindDailyIfOverdue = true` → `notifyReminder = true`

#### 3.5.5 Cronバッチへの影響

現行の `/api/cron/reminders` は `notifyDueToday` と `remindDailyIfOverdue` を個別に判定している。
統合後は `notifyReminder = true` の世帯に対し、期限当日 OR 期限超過のタスクがあれば通知する。

### 3.6 パートナー参加フローの簡素化

#### 3.6.1 現状の課題

現行のパートナー招待フロー:
1. ユーザーAが登録 → 6文字の招待コードが発行される
2. ユーザーAが招待コードをコピー
3. LINEなどで招待コードをユーザーBに送る
4. ユーザーBがアプリを開き、登録画面で招待コードを入力して登録

**問題点**:
- 「家族コード」という概念が直感的でない
- 6文字コードを手動コピー → LINE → 手動入力の手順が多い
- 登録画面の「家族コード（任意）」が目立たず、入力せずに登録してしまうケースがある
- 登録後に参加する場合、設定画面の`JoinHouseholdCard`を見つける必要がある

#### 3.6.2 変更内容

**A. 共有リンク方式の追加**

招待コードに加え、URLリンクでの招待を可能にする。

```
共有リンク形式:
  https://{app-domain}/?invite={inviteCode}

例:
  https://ietasuku.vercel.app/?invite=ABC123
```

- FamilyCodeCardに「リンクをコピー」ボタンを追加（コードコピーと並列）
- 「LINEで送る」等のネイティブ共有（`navigator.share()` API利用）ボタンを追加
- PWAの `navigator.share()` 対応端末ではシェアシートを表示
- 非対応端末ではリンクコピーのみ

**B. リンクからの自動参加**

- アプリ起動時にURLパラメータ `invite` を検出
- 未登録ユーザー: 登録フォームに招待コードを自動入力し、名前入力のみで参加完了
- 登録済みユーザー（別世帯）: 「{inviteCode}の世帯に参加しますか？」確認ダイアログを表示

**C. 登録画面の導線改善**

```
現行:
  名前 [________]
  家族コード（任意）[________]   ← 見落としやすい
  [登録する]

変更後:
  ┌──────────────────────────────┐
  │ はじめての方                  │
  │                              │
  │ 名前 [__________]            │
  │                              │
  │ [はじめる]                    │
  └──────────────────────────────┘

  ┌──────────────────────────────┐
  │ パートナーに招待された方       │
  │                              │
  │ パートナーから届いたリンクか   │
  │ コードで参加できます          │
  │                              │
  │ コード [__________]          │
  │ 名前   [__________]          │
  │                              │
  │ [参加する]                    │
  └──────────────────────────────┘
```

- 2つのパスを明確に分離する
- 「招待された方」を独立セクションにし、見落としを防ぐ

#### 3.6.3 API変更

既存API（`POST /api/register`）で対応可能。`inviteCode`パラメータの有無で分岐する現行ロジックを維持。

### 3.7 ホームウィジェット対応

#### 3.7.1 現状

現行はPWA（`display: standalone`）としてインストール可能だが、ホーム画面ウィジェットは未対応。

#### 3.7.2 技術調査結果

| プラットフォーム | ウィジェット技術 | PWAからの対応 | 備考 |
|-----------------|-----------------|--------------|------|
| Android | App Widgets | 不可（ネイティブのみ） | TWA化してもウィジェットはJava/Kotlin必須 |
| iOS | WidgetKit | 不可（Swift/SwiftUIのみ） | PWAからは一切アクセス不可 |
| Web (実験的) | Widgets API (W3C Draft) | 未実装 | Chrome/Edgeで提案段階、実用化時期未定 |

#### 3.7.3 実現可能な代替策

**A. PWAショートカットアイコン（manifest `shortcuts`）**

```json
// manifest.webmanifest に追加
{
  "shortcuts": [
    {
      "name": "今日の家事",
      "short_name": "今日",
      "description": "今日やるべき家事を確認",
      "url": "/?tab=home",
      "icons": [{ "src": "/shortcut-today.png", "sizes": "96x96" }]
    },
```

- Androidでアプリアイコン長押し時にショートカットメニューが表示される
- 「今日の家事」で直接ホームタブを開ける
- iOSは非対応

**B. プッシュ通知の充実（擬似ウィジェット）**

ウィジェットの代替として、通知内容をリッチにすることで「通知を見れば今日やることが分かる」状態を作る。

```
現行の通知:
  タイトル: 家事リマインド
  本文: トイレ掃除、食器洗い、ほか1件

変更後:
  タイトル: 今日の家事（3件）
  本文:
    🚽 トイレ掃除（2日超過）
    🍽 食器洗い
    🧹 掃除機
```

- タスク名にアイコン絵文字を付与（Lucideアイコン名から絵文字へのマッピングテーブルを用意）
- 超過日数があれば付記
- 全件表示（現行の「上位2件 + ほかN件」を廃止）
- ただしOS側の文字数制限があるため、最大5件 + 「ほかN件」に変更

#### 3.7.4 対応方針

- **v2スコープ**: A（PWAショートカット）+ B（通知リッチ化）を実装
- **ネイティブウィジェット**: ストア配布予定なしのため対象外

## 8. 検討事項の決定記録

以下は検討の結果、方針が確定した事項。

| # | 検討事項 | 決定 | 理由 |
|---|---------|------|------|
| 8.1 | リアクションの通知タイミング | **即時通知**。`notifyCompletion` 設定に準拠 | リアルタイムに伝わることが大事。通知OFFの人には送らない |
| 8.2 | やったよログの表示位置 | **独立タブ「きろく」に昇格** | ホーム下部だとスマホで窮屈。独立画面の方が見やすい |
| 8.3 | おうちレポートの期間 | **直近3ヶ月分**を月単位で切替。データは全期間保持 | 振り返りたいが、見れすぎるとプレッシャーになる。3ヶ月が妥協点 |
| 8.4 | 旧統計画面の扱い | **基本廃止**。移行期間中は併存 | 新旧並行は混乱の元。早めに切り替え |
| 8.5 | リアクション種別の拡張性 | **固定4種で十分** | シンプルさ優先。追加要望が出たら後続対応 |

## 9. 非機能要件の変更

### 9.1 パフォーマンス

- おうちレポートのクエリは月次集計のため軽量（現行statsと同等以下）
- リアクションは `ChoreRecord` にリレーションを追加するため、レコード取得時のJOINが増える
  - `records` API は直近200件制限のため影響軽微
  - bootstrapには含めない（やったよログ表示時にlazy load）

### 9.2 データ量

- リアクションテーブル: 1記録につき最大ユーザー数分（現行想定2名 → 最大2行/記録）
- 増加ペースは緩やか（家事記録数に比例）
